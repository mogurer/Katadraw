# KATA-DRAW リリース方針

## 採用方針（2025年3月時点）

**構造見直しを計った上で体験版をリリースし、反復しつつ製品版の準備を進める**

- まず構造見直し（ステージ定義の分離、設定の集約など）を実施
- 体験版（3ステージ程度）をリリース
- フィードバックに基づき反復改善
- 並行して製品版（全ステージ）の準備を進める

---

## 方針一覧

| 方針 | 概要 | リリース回数 |
|------|------|--------------|
| 方針1 | 体験版 → 製品版 | 2回 |
| 方針2 | 構造見直し → 体験版 → 反復 → 製品版 | 2回 |
| 方針3 | いきなり製品版 | 1回 |
| 方針4 | 早期アクセス / ソフトローンチ | 1回（継続更新） |
| 方針5 | 超軽量体験版 → 即製品版 | 2回 |

---

## 方針1: 体験版 → 製品版（2段階）

### 概要

操作性を改善した後、3ステージ程度の体験版をリリース。フィードバックを反映しつつ製品版（全ステージ）を準備する。

### フロー

```
操作性改善 → 体験版リリース → フィードバック収集 → 改善・ステージ追加 → 製品版リリース
```

### メリット

- 早期フィードバックで方向修正しやすい
- 製品版の完成度を上げやすい
- 体験版で認知・興味を集めやすい

### デメリット

- 2回のリリース作業が必要
- 1〜2か月のスケジュールでは時間がタイト

### 向いているケース

- フィードバックを重視してから製品版を出す
- スケジュールに余裕がある

---

## 方針2: 構造見直し → 体験版 → 反復 → 製品版（3段階）

### 概要

まずコード構造を見直し、その後体験版をリリース。フィードバックに基づく反復改善をしやすくし、製品版へと進める。

### フロー

```
構造見直し → 体験版リリース → フィードバック収集・反復改善 → ステージ追加・仕上げ → 製品版リリース
```

### メリット

- フィードバック対応が速い
- 製品版開発が効率化される
- 長期的な保守性が上がる

### デメリット

- 構造見直しに1〜2週間かかる
- 最初のリリースまでが遅くなる
- 1〜2か月のスケジュールでは特に厳しい

### 向いているケース

- フィードバックを前提にどんどん改善したい
- スケジュールに余裕がある

---

## 方針3: いきなり製品版（1段階）

### 概要

体験版を出さず、最初から製品版（全ステージ）としてリリースする。

### フロー

```
操作性・品質の最終調整 → 製品版リリース
```

### メリット

- リリースは1回で済む
- 体験版の切り替え・運用が不要
- スケジュールが単純で管理しやすい
- 「完成品」として出せる

### デメリット

- 事前のフィードバックがない
- 致命的な問題に気づくのがリリース後になる
- バランス調整もリリース後にやり直しになりがち
- 認知・興味づくりを別途考える必要がある

### 向いているケース

- すでに十分プレイ可能で、大きな不安がない
- 小規模・低価格で、リリース後の修正を許容できる
- まず出して、反応を見ながら直す方針でよい

---

## 方針4: 早期アクセス / ソフトローンチ（製品として出しつつ継続改善）

### 概要

最初から「製品」として販売するが、「早期アクセス」「開発中」と明示。現時点の内容でリリースし、フィードバックに応じて更新しつつ、完成度が上がったら「1.0」として正式リリースする。

### フロー

```
現状の内容でリリース（早期アクセス） → フィードバック収集・更新 → 継続改善 → 1.0 正式リリース
```

### メリット

- リリースは1回で済む
- フィードバックを得ながら改善できる
- 価格を抑えれば「開発中」として許容されやすい
- Steam などでは「早期アクセス」の枠で出せる

### デメリット

- 「早期アクセス」の期待値に応える必要がある
- 更新頻度や品質維持の負担が続く

### 向いているケース

- Steam などのプラットフォームを利用する
- 販売しながら継続改善したい

---

## 方針5: 超軽量体験版 → 即製品版（短期集中）

### 概要

2〜3週間で「最低限プレイできる3ステージ」を出す。構造見直しは最小限（例: ステージ定義の分離だけ）にとどめ、1〜2週間フィードバックを集める。残り期間で残りステージ追加・調整して製品版としてリリースする。

### フロー

```
最小限の構造見直し → 体験版リリース（2〜3週間） → フィードバック収集（1〜2週間） → ステージ追加・調整 → 製品版リリース
```

### メリット

- フィードバックを短期間で得られる
- 構造の大規模リファクタを避けられる
- 1〜2か月のスケジュールに収まりやすい

### デメリット

- 体験版の品質が粗くなりがち
- フィードバック反映の時間が限られる

### 向いているケース

- 1〜2か月でリリースしたい
- スピードとフィードバックのバランスを取りたい

---

## 採用方針の詳細（構造見直し → 体験版 → 反復 → 製品版）

方針2と方針5の要素を組み合わせた形となる。

### フロー

```
構造見直し（1〜2週間）
    ↓
体験版リリース（3ステージ）
    ↓
フィードバック収集・反復改善（並行）
    ↓
製品版の準備（ステージ追加・仕上げ）
    ↓
製品版リリース
```

### 構造見直しで実施する項目

- ステージ定義の分離（外部リソース・設定化）
- 設定・定数の集約（ヒントタイミング、クリア条件など）
- 体験版と製品版の切り替え方法の整備（例: ステージ数の設定）

### 体験版の範囲

- ステージ数: 3ステージ程度
- 体験版終了時の画面・文言の検討

### スケジュール目安（1〜2か月）

| 期間 | 内容 |
|------|------|
| 1〜2週間 | 構造見直し |
| 2〜3週間 | 体験版リリース・フィードバック収集 |
| 残り期間 | 反復改善・製品版準備・リリース |
