# KATA-DRAW 調査・改善点整理

## 1. プロジェクト構造の把握

### 1.1 全体構成

```
Katadraw/
├── project.godot          # プロジェクト設定（Godot 4.6、1920x1080）
├── scenes/
│   └── game.tscn          # メインシーン（Node2D + game.gd）
├── scripts/
│   ├── game.gd            # オーケストレーター（状態遷移・入力振り分け・約1100行）
│   ├── stage_manager.gd   # ステージ生成・メトリクス・クリア判定
│   ├── input_handler.gd  # マウス・パッド・ドラッグ・BB
│   ├── stage_renderer.gd # 図形タイプ別描画
│   └── ui_renderer.gd    # UI・パーティクル・エフェクト
├── Resources/
│   ├── stage_data.gd      # ステージ定義
│   ├── stage_config.gd    # ステージデフォルト設定
│   └── game_config.gd    # ゲーム設定
├── assets/
│   ├── UI/                # ロゴ画像
│   ├── sounds/            # BGM・SE
│   └── fonts/             # DIN1451alt
└── ...
```

### 1.2 アーキテクチャの特徴

- **モジュール分離済み**: game.gd（オーケストレーター）、StageManager、InputHandler、UIRenderer、StageRenderer に責務を分離
- **シーン構成**: `game.tscn` は Node2D に `game.gd` をアタッチした最小構成
- **描画**: `_draw()` 内で CanvasItem の draw_* 系を直接使用
- **入力**: `_input()` でイベントを一元処理

## 2. ゲームフロー（状態遷移）

```
logo → title → menu → rules → guide_info → guide_countdown → playing
                                                                    ↓
title ← results ← cleared ← playing
         ↑
    config（menu から分岐）
    pause（playing 中に ESC/Start でオーバーレイ）
```

| 状態 | 説明 |
|------|------|
| logo | 起動ロゴ表示（7秒で自動遷移） |
| title | タイトル画面 |
| menu | ゲームスタート / コンフィグ選択 |
| config | 解像度・ウィンドウモード・言語 |
| rules | 操作方法説明 |
| guide_info | ステージ開始前のお手本表示 |
| guide_countdown | 3秒カウントダウン |
| playing | 本編プレイ |
| cleared | ステージクリア |
| results | 全ステージクリア後のリザルト |

## 3. ステージ構成

7ステージが `STAGES` 定数で定義：

| # | タイプ | 点数 | 内容 |
|---|--------|------|------|
| 1-5 | circle | 12〜20 | 正円に近づける（難易度上昇） |
| 6 | two_circles | 24 | 2つの正円（各12点） |
| 7 | star | 20 | 五芒星に近づける |

**クリア条件**: 再現率（circularity）と滑らかさ（smoothness）が目標値以上（93〜97%）

## 4. 主要機能の実装概要

### 4.1 入力システム

- **マウス**: ドラッグ（ポイント移動）、ホバーで選択
- **ゲームパッド**: 左スティック（選択切り替え/ポイント移動）、A（つかむ/離す）、LB/RB（ポイント切り替え）、右スティック（ポイント選択・つかみ）、十字キー（隣接切り替え/ポイント移動）
- **入力ロック**: デモ画面で「次へ」を選んだ操作（マウス or パッド）にプレイ中の入力をロック。詳細は `Docs/特殊操作一覧.md`

### 4.2 メトリクス計算

- **円**: 重心からの距離のばらつき（circularity）、隣接点間の滑らかさ（smoothness）
- **星**: 5角形へのフィット誤差（回転・スケールを探索）

### 4.3 その他

- バウンディングボックスによる複数選択の一括変形（8アンカー+中心）
- パーティクル演出（クリア時）
- ヒント表示（60秒・90秒、120秒以降はループ）

## 6. 現時点で対応すべき課題

体験版リリースに向け、直近で対応すべき項目：

| 項目 | 内容 |
|------|------|
| **操作性の見直し** | マウス・ゲームパッドの操作感、感度、ヒット判定などの調整 |

## 7. 改善点の整理

### 7.1 アーキテクチャ・保守性

| 優先度 | 項目 | 内容 |
|--------|------|------|
| ～ | 単一ファイル肥大化 | **対応済み**: StageManager, InputHandler, UIRenderer, StageRenderer に分離 |
| 中 | 定数・設定の外部化 | STAGES、色、タイミングなどをリソースや設定ファイルに分離 |
| 中 | シーン分離 | タイトル、メニュー、プレイ画面などを別シーン化し、シーン遷移で管理 |

### 7.2 コード品質

| 優先度 | 項目 | 内容 |
|--------|------|------|
| 中 | マジックナンバー | 座標（例: `vp.x * 0.25`）、フォントサイズ、マージンなどが散在。定数化 |
| 中 | 重複コード | `_draw_rules` と `_draw_pause_overlay` の操作方法表示がほぼ同一。共通化 |
| 低 | 型アノテーション | 一部で型が省略されている。明示して可読性・安全性を向上 |

### 7.3 機能・UX

| 優先度 | 項目 | 内容 |
|--------|------|------|
| 中 | 言語の初期値 | `_ready()` で `TranslationServer.set_locale("ja")` を固定。コンフィグの保存値を使う |
| 中 | ゲームパッドの初期化 | `pad_cursor_initialized` が playing 時のみ。メニューから playing に入る前に初期化する方が自然 |
| 低 | ヒントの表示時間 | 60秒・90秒は長い。初回プレイ向けに短縮や段階表示を検討 |
| 低 | リトライ | ステージ失敗時のやり直し機能がない（現状はクリアのみ進行） |

### 7.4 パフォーマンス・最適化

| 優先度 | 項目 | 内容 |
|--------|------|------|
| 低 | queue_redraw の頻度 | 多くの状態で毎フレーム `queue_redraw()`。変化時のみ呼ぶように最適化可能 |
| 低 | フォント読み込み | 毎回 `_ready` で読み込み。キャッシュやプリロードの検討 |

### 7.5 潜在的な不具合

| 優先度 | 項目 | 内容 |
|--------|------|------|
| ～ | two_circles の選択クリア | **対応済み**: group1/2 クリア時に `input_handler.release_mouse_grab()` でマウス状態を解除 |
| 低 | ビューポート変更 | 解像度変更時に `shape_center` 等が再計算されない可能性。リサイズ時の再計算を検討 |

### 7.6 ドキュメント・運用

| 優先度 | 項目 | 内容 |
|--------|------|------|
| 中 | README | プロジェクト概要、起動方法、ビルド手順を記載 |
| 低 | コメント | 複雑な計算（例: `_star_distance_at_angle`）に説明コメントを追加 |

---

## 8. 推奨アクション（優先順）

1. **`game.gd` の分割**: ステージ管理・入力・UI・オーディオなど、責務ごとにクラス/スクリプトへ分離
2. **定数・設定の外部化**: `STAGES` や色定義をリソース化
3. **言語設定の永続化**: コンフィグで選んだ言語を保存し、起動時に復元
4. **操作方法表示の共通化**: `_draw_rules` と `_draw_pause_overlay` の共通部分を関数化
5. **ウィンドウリサイズ対応**: 解像度変更時に `shape_center` などを再計算

## 9. 補足

- ゲームの核となる「点の操作→形の一致」のロジックは明確で、メトリクス計算も妥当
- マウス・ゲームパッド両対応、多言語、コンフィグなど、完成度は高い
- 改善は主に保守性・拡張性の向上と、細かいUX調整に集中すると効果的
